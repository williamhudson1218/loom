# Chat Manager — Design

**Date:** 2026-06-17
**Status:** Approved, pre-implementation
**Location:** `~/.claude/tools/chat-manager/` (standalone, outside any project repo)

## Problem

Will runs ~20 concurrent Claude Code sessions across ~9 projects (hundreds of
session transcripts on disk — 379 in tax-pilot-app alone). Two pain points:

1. **Chats are unmanageable.** Claude's auto-titles are stale/useless, the resume
   list is too shallow to find anything, and there's no at-a-glance view of what
   each chat was about or how recently it was touched. `/find-chat` exists but is
   built for one-off *search*, not ongoing *management*.
2. **"Lost" chats after `/clear`.** Will believed `/clear` was overwriting chats.
   **Verified false:** all 379 sessions are on disk, one session per file, each
   file's internal `sessionId` matches its filename, no file holds >1 session.
   `/clear` starts a *fresh* session and leaves the old one untouched — it just
   becomes invisible. This is a **discoverability** problem, not data loss. The
   dashboard fixes it by surfacing every session.
3. **tmux crashes ~weekly**, destroying the whole multi-session / multi-pane
   workspace (and the running Claude chats inside it). There is no way to rebuild
   it or to get the chats back into their panes.

## Goals

- A continuously-maintained, human-readable index of **every** Claude Code
  session across **all** projects, with genuinely useful AI-generated
  `title / overview / breakdown`, a recency/activity view, and one-click resume.
- Track which **tmux session + pane** each chat lives in, and snapshot the full
  tmux layout, so the entire workspace (geometry + chats + programs) can be
  rebuilt after a crash.

## Non-goals

- Not a live server / web app (static regenerated HTML only).
- Not modifying or replacing `/find-chat` (separate tool, separate DB).
- Not generic tmux session management beyond snapshot + restore (no tmux-resurrect
  replacement; we add the Claude-resume layer on top of a self-contained snapshot).

## Key decisions (from brainstorming)

| Decision | Choice | Why |
|---|---|---|
| Build vs extend `find-chat` | **New separate tool** | Isolated; cannot break `/find-chat`. Accepts duplicating the parser. |
| Summary engine | **Headless Claude Code agent** (`claude -p`) | Uses existing Claude Code login (no API key); best-quality summaries. |
| UI surface | **Regenerated static `dashboard.html`** | Dead simple, no daemon serving traffic. |
| Activity chart | **Both** — global heatmap banner + per-chat recency sparkline | At-a-glance "how busy" + per-chat hot/stale. |
| Scope | **All projects** under `~/.claude/projects` (~9 dirs) | Whole workspace, not just tax-pilot. |
| Cold-start cost | **Cap ~25 chats summarized per pass** | First run drains gradually over hours, not one burst. |
| tmux restore scope | **Full layout, smart relaunch** | Closest to real workspace: exact geometry, Claude auto-resume, nvim relaunch, others pre-typed (not auto-run). |
| Ghostty tabs | **Default: rebuild tmux + print/clipboard attach list. Opt-in `--fresh-tabs` AppleScript** | tmux crash leaves Ghostty tabs alive → AppleScript would create duplicate tabs. AppleScript only for cold-start (no tabs); brittle + needs Accessibility, so kept optional. |

## Phasing

- **Phase 1 — Dashboard** (self-contained, ships first): parser → indexer →
  analyzer → dashboard generator, scheduled via launchd.
- **Phase 2 — Workspace placement & restore**: SessionStart hook, tmux snapshot,
  `chat-manager restore` (+ optional `--fresh-tabs`).

The schema is designed up front to support both phases so Phase 2 adds tables/
columns without reworking Phase 1.

---

## Architecture

Four decoupled, individually-testable stages plus a CLI and a launchd job. One
SQLite DB.

```
~/.claude/tools/chat-manager/
  src/parser.ts      JSONL -> {sessionId, project, timestamps, msgs, files, firstMsg, autoTitle, activityByDay}
  src/indexer.ts     scan all projects, mtime-gated; upsert raw rows; mark summary_dirty
  src/analyzer.ts    headless `claude -p` over dirty rows -> {title, overview, breakdown[]}
  src/snapshot.ts    [Phase 2] capture live tmux layout; join panes to session_ids via placements
  src/dashboard.ts   read DB -> write self-contained dashboard.html
  src/restore.ts     [Phase 2] replay latest layout snapshot
  src/cli.ts         `chat-manager run` (index -> analyze -> [snapshot] -> render); `chat-manager restore [--fresh-tabs]`
  src/db.ts, src/schema.ts, src/paths.ts, src/types.ts
  manager.db         SQLite (own schema)
  dashboard.html     regenerated each pass
  com.chatmanager.plist  launchd job -> `chat-manager run` every ~10 min
  hooks/record-placement.sh  [Phase 2] SessionStart hook
```

The launchd job only calls `chat-manager run`. Each stage is wrapped so one bad
transcript cannot abort a pass.

## Data flow — one `run` pass

1. **Index.** Walk every project dir. For each JSONL whose mtime > stored mtime,
   parse and upsert a raw row (timestamps, message_count, per-day activity counts,
   files touched, first user message, Claude's auto-title). Set `summary_dirty=1`.
2. **Analyze.** For each `summary_dirty` row (capped ~25/pass, concurrency ~3),
   spawn `claude -p` with the transcript and a strict prompt → JSON
   `{title, overview, breakdown[]}`. Write fields, clear `summary_dirty`. Chats
   with <3 messages get a heuristic title only (skip the LLM).
3. **Snapshot** *(Phase 2).* `tmux list-panes -a` → capture every session/window/
   pane (geometry `window_layout`, cwd, command, title, kind). For each pane look
   up `pane_id` in `placements` to attach a resumable `session_id`. Persist as one
   `layout_snapshots` row. Update each chat's `last_tmux_session` / `last_pane_id`.
4. **Render.** Regenerate `dashboard.html` from the DB.

Re-analysis triggers only when a transcript's mtime changes (i.e. the chat grew,
including on resume), so untouched chats are never re-billed.

## Database schema (`manager.db`)

```sql
-- Phase 1
chats(
  session_id PK, project_dir, jsonl_path,
  started_at, ended_at, last_active_at, message_count,
  activity_json,        -- {"2026-06-15": 12, ...} per-day msg counts
  files_touched,        -- newline-separated
  first_message,
  claude_auto_title,    -- Claude's stale title (kept for comparison)
  title, overview,      -- AI-generated by analyzer
  breakdown_json,       -- ["did X", "decided Y", "left off at Z"]
  summary_dirty, summary_model, summary_at,
  last_tmux_session, last_pane_id,   -- populated in Phase 2
  jsonl_mtime, last_indexed_at
);

-- Phase 2
placements(           -- live pane_id -> session_id, written by the SessionStart hook
  session_id PK, tmux_session, window_index, pane_index,
  pane_id, cwd, recorded_at
);

layout_snapshots(     -- one row per pass; the restorable blob
  taken_at PK,
  snapshot_json        -- [{session, window_layout, panes:[{pane_index, cwd,
                       --   command, title, kind:'claude'|'nvim'|'shell', session_id?}]}]
);
```

## Phase 1 detail — analyzer prompt contract

`claude -p` is fed the transcript (head+tail truncation for very large files to
bound tokens) and must return **only** JSON matching a fixed schema:

- `title` — ≤ 8 words, specific (not "Claude Code").
- `overview` — ≤ 25 words, one sentence.
- `breakdown` — 3–6 past-tense bullets of what happened, the **last** bullet being
  where the chat left off.

Parse failure / non-JSON → leave `summary_dirty=1`, log, retry next pass.

## Phase 1 detail — dashboard (`dashboard.html`)

Self-contained HTML with an embedded JSON blob; all interactivity is client-side
JS (no server, no network).

- **Top banner:** GitHub-style 12-week activity heatmap aggregated from all chats'
  `activity_json`.
- **Controls:** text filter + project filter + sort (recent / longest / most
  active), operating on the embedded JSON.
- **Cards**, newest-active first, each: AI **title**; recency **sparkline**
  `▁▃▇█▅` + "2h ago"; one-line **overview**; expandable **breakdown** bullets;
  project + message count; **session id with copy button** and the exact
  `cd <dir> && claude --resume <id>` line; [Phase 2] `📍 taxpilot · pane 1`.
- **Empty-summary state:** chats indexed but not yet summarized show "indexed,
  summary pending" (honest, never blank).

## Phase 2 detail — the pane↔session link (the crux)

Reverse-engineering "which pane = which chat" by observation is **not reliable**:
the transcript file is not held open continuously (no stable lsof fd), and the
`claude` process exposes no session id in argv or env (verified on the live box).

Two cooperating sources, joined by tmux's stable `pane_id` (e.g. `%9`):

1. **`SessionStart` hook** (`hooks/record-placement.sh`) — fires on every chat
   **start and resume**. Reads `session_id` from stdin JSON and `$TMUX_PANE` from
   env, resolves its own session/window/pane/cwd via `tmux display-message`, and
   upserts one `placements` row. Authoritative `session_id → pane_id` link.
   Replaces the existing racy `/tmp/claude-tmux-pane-latest` approach (which keyed
   off the wrapper PID, not the session UUID).
2. **Periodic tmux snapshot** (`snapshot.ts`, same pass) — captures the whole live
   layout including non-Claude panes (nvim, prod tunnels, `node` dev servers). For
   each pane, looks up `pane_id` in `placements` to attach the `session_id`.

`pane_id` is stable within a tmux server lifetime, and both the hook and the
snapshot run within that lifetime, so the join is valid. After a crash the server
dies and pane ids reset — but by then the last good snapshot (with resolved
session ids) is already on disk.

A command-based heuristic (`pane_current_command` ∈ {version string, `node`} +
title) is a fallback for panes not yet in `placements`.

## Phase 2 detail — `chat-manager restore`

Reads the latest `layout_snapshots` row and rebuilds it:

- `tmux new-session -d -s <name> -c <cwd>` per session; recreate panes; then
  `select-layout "<window_layout>"` to restore exact geometry.
- **Claude panes** → `send-keys "claude --resume <session_id>" Enter` (matching
  Will's `--dangerously-skip-permissions` flow).
- **nvim panes** → relaunch `nvim` in the saved cwd.
- **Other panes** (tunnels, dev servers) → shell in the saved cwd with the last
  command **pre-typed but not executed** (so a tunnel/server never auto-fires).

**Ghostty attach:**

- **Default:** after rebuilding, print the `ta <name>` attach list and copy it to
  the clipboard. In the common crash case (tmux server died, Ghostty tabs survive
  as dead shells) Will pastes into the existing tabs — reliable, no permissions,
  no duplicate tabs.
- **`--fresh-tabs` (opt-in, cold start only):** drives Ghostty via System Events
  AppleScript (⌘T per session, type `ta <name>`) to recreate tabs when none exist
  (reboot / Ghostty closed). Best-effort; falls back to the printed list if
  Accessibility permission is absent. Ghostty's CLI can open a window but has no
  clean "open tab running a command", so AppleScript is the only full-auto path —
  hence kept optional and last.

## Error handling & safety

- **Read-only** on all transcripts; the tool writes only its own `manager.db` and
  `dashboard.html`.
- Per-pass analyze cap (~25) + concurrency limit (~3): never stampedes the machine
  or token budget.
- Every stage wrapped: one bad transcript/pane can't abort the pass.
- `chat-manager run` is idempotent; launchd `StandardError` → a log file.
- `restore` never kills existing tmux sessions; it only creates. Non-Claude
  commands are pre-typed, not executed.

## Testing

- `parser.ts` + activity/sparkline math — pure unit tests on fixture JSONLs.
- `indexer.ts` — mtime-gating (unchanged file skipped; grown file re-marked dirty).
- `analyzer.ts` — prompt-builder + JSON-response parser, tested with a **stubbed**
  `claude` runner (no real API/LLM in tests).
- `dashboard.ts` — snapshot test: given N rows, emits valid HTML with the right
  cards and an embedded JSON blob.
- [Phase 2] `snapshot.ts` — pane→session join from fixture `tmux list-panes`
  output + a fixture `placements` table.
- [Phase 2] `restore.ts` — given a fixture snapshot, asserts the exact sequence of
  `tmux` commands it would emit (dry-run, no real tmux).
